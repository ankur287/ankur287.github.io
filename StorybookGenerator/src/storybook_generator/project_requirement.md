# Product Requirements Document: Children's Storybook Generator

**Version:** 1.0  
**Status:** North Star — Governing Document  
**Last Updated:** 2026-05-07

---

## What This Product Is (Deployment Model)

**This is a local laptop application, not a cloud service.**

- **The Application:** A Python process that runs a lightweight web server on the laptop (`localhost:PORT`)
- **The Interface:** A web UI opened in any browser on the same machine (no hosting, no public URL)
- **The Models:** Both the text generator (Ollama + open LLM) and image generator (Stable Diffusion via `diffusers`) run entirely on the laptop
- **The Output:** Self-contained HTML storybook files saved to a local folder
- **Optional:** Those HTML files can be manually pushed to `ankur287.github.io` to archive or share, but this is not part of the core workflow
- **No Internet Required:** After downloading the models once, the entire application works completely offline

**Why this model?** It's the simplest possible option — no hosting costs, no cloud API fees, no data leaving the device, no internet dependency after setup, no third-party accounts. It aligns with the original requirement for "local models" and preserves complete privacy for children's data.

## Vision & Purpose

### What This Product Is

The Children's Storybook Generator is a personal, locally-run tool that transforms a parent or caregiver's imagination into a beautifully illustrated, fully interactive children's storybook. The user describes a story idea and sets the child's age; the application generates a complete, age-appropriate story with a meaningful moral, illustrates every page with cartoon artwork, and presents the result as a digital storybook with physical page-turn animations — something a child can sit down and "read" on a screen.

### Who It Is For

The primary user is an adult — a parent, grandparent, aunt/uncle, or teacher — who wants to create a personalized story for a specific child. The child is the audience, not the operator. The child's name, age, favorite animals, interests, or personal situations can feed into the story. The operator is assumed to be technically comfortable enough to run a local Python application but is not expected to understand machine learning or prompt engineering.

### Why It Exists

Personalized children's stories are one of the most powerful ways to engage a child's imagination, reinforce values, and address specific life situations (a new sibling, starting school, dealing with fear of the dark). Commercial story apps are generic. Hiring an illustrator is expensive. Cloud AI tools raise privacy concerns when children's data is involved and require internet connectivity.

This tool exists so that anyone, at home, offline, without sending data to any third party, can generate a deeply personalized and beautifully presented story for a child they love.

### The Core Promise

Every story generated by this tool must feel like it was made for that specific child, look like a real picture book, and be something the adult is proud to sit down and read aloud together.

## User Journey

The complete user journey from opening the application to reading a story follows these stages:

**Stage 1 — Landing on the Home Page.** The user opens the application in their browser (served locally at `localhost`). They see two options: **"Create New Story"** and **"My Stories."** If they have generated stories before, thumbnails or title cards of past stories are visible in the My Stories section, arranged like books on a shelf. The page feels warm and inviting — not like a developer tool or dashboard.

**Stage 2 — Filling In the Story Form.** The user clicks "Create New Story" and is taken to a form. The form is conversational in tone, not clinical. Each field has a label and a short explanation of why it matters. The user fills in details about the child and the story. There is a single **"Generate Story"** button at the bottom. There are no advanced settings, no model parameters, no technical options visible.

**Stage 3 — Generation In Progress.** After submitting, the user sees a waiting screen. This screen is not a blank spinner — it shows a friendly message like **"Writing your story..."** followed by **"Now drawing the pictures..."** as generation progresses. The wait is expected to be 2–10 minutes depending on laptop hardware. The user should feel confident something is happening, not wonder if the app crashed.

**Stage 4 — Reading the Storybook.** When generation completes, the storybook opens immediately. The user sees the cover page. They can tap or click an arrow to turn pages. Pages turn with a physical page-curl animation. Each page displays one illustration at the top, the story text below it in a storybook font, and the page number at the bottom. The final page shows the moral/lesson in a slightly different typographic treatment to signal it is the takeaway.

**Stage 5 — Returning to the Library.** After reading, the user can return to the home page. The story is already saved in the library. On future visits, the user can find it by its title and cover thumbnail, click it, and it opens back to the cover page ready to read again.

## Input Specification

The form collects exactly the following inputs. Every field except Story Topic and Child's Age is optional, but the UX should encourage completion because richer input produces better stories.

### Required Inputs

**Story Topic (required).**  
A free-text field. This is what the story is about. Examples: "a shy bunny who learns to make friends," "a girl who is nervous about starting school," "a boy named Leo who loves dinosaurs goes on an adventure." Label: "What's your story about?" Users are explicitly told: *write it like you're describing the story idea to a friend.* Minimum 10 characters, maximum 300 characters.

**Child's Age (required).**  
A numeric input or simple selector: **2, 3, 4, 5, 6, 7, 8, 9, 10+** years. This is the single most important calibration input. It governs vocabulary complexity, sentence length, story length, the nature of the moral, and the emotional register of the illustrations. Label: "How old is the child?" The system must never ask the user to think about how age maps to content — that is the system's job.

### Optional Inputs (but encouraged)

**Child's Name (optional).**  
Label: "Child's name (so they feel special)." If provided, the main character will share this name, or the story will reference the child by name in a way that feels personal. Maximum 40 characters. Must be sanitized — the system will reject names that are offensive or that would break the narrative.

**Main Character (optional).**  
Label: "Who or what is the story about?" Free text. Examples: "a red panda," "a little girl who loves painting," "a robot who wants to feel emotions." If blank, the system invents a character appropriate to the story topic and age.

**Story Theme / Mood (optional).**  
Label: "What feeling should the story have?" A small set of preset options: **Adventurous, Funny, Gentle and Calm, Magical, Heartwarming.** Defaults to "Heartwarming" if not chosen. This steers the tone of both writing and illustration style.

**Moral or Lesson (optional).**  
Label: "What should your child learn?" Free text, up to 100 characters. Examples: "being kind to strangers," "it's okay to ask for help," "differences make us special." If blank, the system infers the most natural moral from the story topic. This field uses plain language for non-technical users, not the word "moral."

**Dedicated To (optional).**  
Label: "Make it a gift?" A name or short phrase like "For Lily, who makes every day brighter." If provided, this appears on a dedication page (the second page, after the cover). Maximum 80 characters.

### What Is NOT an Input

No other inputs are accepted. There is no field for "number of pages," "illustration style," "language," "model settings," or "reading difficulty." Those are implementation decisions, not user decisions.

## Story Content Rules

### Age Tiering

The system maps the provided age to one of three internal tiers. The developer should use these tiers as the governing constraint for all story generation prompts.

**Tier 1: Ages 2–4 (Early Listeners)**  
The child is not yet reading; a parent reads aloud. Stories use simple, repetitive sentence structures. Vocabulary is limited to common, concrete words — no abstractions. Story length: 6–8 pages of body content plus cover. Each page has at most 3 sentences. The moral is concrete and demonstrated through action, never stated as an abstract principle. Example moral delivery: "And from that day on, Benny always shared his acorns with his friends."

**Tier 2: Ages 5–7 (Early Readers)**  
The child may read simple words or follow along. Sentences can be slightly longer. Vocabulary can include vivid descriptive words but should avoid idioms or cultural references the child would not know. Story length: 10–12 pages of body content plus cover. Each page has 3–5 sentences. The moral can be slightly more explicit but should still be primarily shown through the character's experience.

**Tier 3: Ages 8–10 (Independent Readers)**  
The child reads independently. Sentences can be complex. Vocabulary can be rich. The story can have a B-plot or minor twist. Story length: 12–16 pages of body content plus cover. The moral can be stated or reflected on by the character at the end.

### Fixed Story Structure

Every generated story follows this fixed 7-part structure, regardless of tier:

1. **Cover Page** — Title and one establishing illustration. No body text.
2. **Dedication Page** (if a dedication was provided) — Short dedication text, decorative but no illustration.
3. **Opening Pages** — Sets the scene and introduces the main character. Conveys the character's world and the central challenge or desire.
4. **Rising Action Pages** — The character tries, struggles, or goes on a journey. Each page advances the situation. In Tier 1, this is simple repetition with variation ("She tried again..."). In Tier 3, there can be a complication.
5. **Climax Page** — A single page where the central moment of change, discovery, or resolution occurs.
6. **Resolution Pages** — The world after the change. Warm, settled, hopeful.
7. **Final Page / Moral Page** — The story's takeaway. In Tier 1, this is shown through action. In Tier 3, the character can reflect. Typographically distinct from body pages.

The generation prompt must explicitly reference these 7 parts so that the output maps cleanly onto numbered pages in the HTML.

### Quality Requirements for Story Text

Every generated story must satisfy all of the following before being assembled into the final HTML:

- Every sentence is grammatically correct and complete.
- No page ends mid-sentence.
- The character's name is consistent throughout (exactly as entered, capitalized correctly).
- The moral or lesson is present and legible on the final page.
- No character in the story uses, recommends, or is associated with real-world brands, commercial products, or copyrighted fictional characters.
- The emotional register is warm and hopeful. No story ends in unresolved sadness, fear, or punishment without redemption.
- There are no pronouns or descriptions that assume gender unless the user has specified a gendered character.
- The story reads as if a thoughtful human wrote it for this specific child, not as if a machine wrote it.

### Age-Appropriate Content: What Is Prohibited

The system must enforce the following prohibitions at the content generation level, regardless of what the user requests:

- **No violence of any kind,** including cartoon or consequence-free violence. This includes fighting, hitting, war games, or aggressive conflict.
- **No death,** even of minor characters. (Illness, loss, or separation may be addressed only in Tier 3 and only as a gentle thematic backdrop, never as a scene.)
- **No scary imagery designed to frighten.** Mild adventure tension is acceptable in Tier 2 and 3 (a character getting temporarily lost, missing a friend), but not imagery intended to distress.
- **No romantic or sexual content** of any kind.
- **No mockery, cruelty, or bullying** portrayed as acceptable or fun.
- **No real political figures, living public figures, or religious figures** as characters.
- **No substance abuse, illegal activity, or criminal behavior** portrayed as acceptable or fun.

### Age-Appropriate Content: What IS Permitted

The safety system must not be over-aggressive. The following are explicitly permitted and the system must not block them:

- Mildly suspenseful or adventurous situations resolved positively (a child temporarily lost in the woods who finds their way home).
- Conflict between characters that is resolved through kindness, understanding, or cleverness.
- Characters making mistakes and learning from them.
- Themes of sadness, loneliness, or missing someone that are processed and resolved (a character misses a friend who moved away but makes a new one).
- Any cultural background, name, origin, or setting from any real-world culture.
- Environmental themes like caring for animals or protecting nature.

## Visual & UX Design Principles

### The Storybook Feel Is Non-Negotiable

The entire visual experience must evoke holding a physical picture book. This is not a website with text content. It is not a slideshow. It is a book. Every design decision — typography, color, animation, spacing — must be evaluated against the question: **does this feel like a children's book?**

### The Page Turn Animation

The page turn is the product's signature interaction. It must use a **CSS 3D transform-based page curl animation** — not a slide, not a fade, not a flip. The animation should simulate a right-to-left page turning, with the page appearing to have thickness and the back of the page briefly visible during the turn. Animation duration should be 600–900ms. It must feel satisfying and slightly physical, like turning a page in a real book.

**Interaction:** Tapping the right half of the page or clicking the right arrow advances the story; tapping/clicking the left half or left arrow goes back. Keyboard arrow keys must also work. The animation should feel smooth on a standard laptop from the last 5 years.

### Typography

**Two font families are used across the entire application, always:**

**Story Text Font:** A rounded, warm serif or semi-serif typeface that evokes handwritten warmth but remains highly legible. Google Font suggestions: "Patrick Hand," "Nunito," "Caveat" (display only), or similar. Body text must be large enough that a child following along can read it — **minimum 18px at 1280px viewport width**, scaling proportionally down.

**UI Font (non-book elements):** A clean, friendly sans-serif for the application chrome (form labels, navigation, library titles). Never use system UI fonts or anything that looks corporate.

**Every page has the story title as a header or footer, helping the reader orient within the book.**

### Color and Backgrounds

**Each story gets a single cohesive color palette** derived from its theme and generated at creation time. This palette is applied consistently to every page background, decorative elements, and typography color. The palette must be warm, saturated but not garish — think watercolor picture book palettes, not primary web colors.

**Page backgrounds are not solid colors.** They use a subtle illustrated or watercolor-texture pattern that matches the story's setting — a forest scene at the edges of a forest story, star patterns for a space story, soft cloud shapes for an adventure story. These are decorative background elements, not full illustrations, and must never compete with the central illustration and text.

### Illustration Placement and Style

**Each body page has one illustration** occupying roughly the top 55–60% of the page. The illustration is horizontally centered with soft rounded corners. It has no hard border — it should feel like the image is part of the page, not pasted onto it.

**Illustrations are in a consistent cartoon/watercolor style throughout a given story.** Style parameters are set once per story (not per image) and must produce visual coherence — characters must look recognizably the same from page to page.

**The cover illustration** is full-page (or near full-page), with the title overlaid on it in a typographically prominent treatment.

### The Application Frame (Outside the Book)

The application chrome must be **minimal and warm**, never clinical. 

- **Home page:** Stories are displayed as book spines or cover thumbnails arranged like books on a shelf, not as a list or grid.
- **Creation form:** Clean, centered, with generous whitespace.
- **Overall color:** A warm off-white or very light cream, never stark white.
- **Mobile:** On mobile screens, the storybook displays one page at a time in portrait mode (the default reading orientation). On desktop, single-page display is the default; two facing pages (spread view) may optionally be shown for Tier 3 stories to enhance the physical book feel.

## Safety & Content Moderation

### Principle

**Safety is a hard constraint, not a best-effort feature.** The system must never produce content that could harm, frighten, or expose a child to age-inappropriate material — even if the user explicitly requests it. The operator is trusted; the input is not.

### Input Screening

Before any generation begins, the story topic, character name, moral/lesson, and dedication text are screened for inappropriate content. Screening is performed by the same local LLM used for story generation, using a **dedicated classification prompt that runs separately and before story generation begins.** The classification must check for:

- Sexual content of any kind
- Graphic violence or gore
- Content that demeans a protected group of people
- Requests to portray real, named individuals in fictional scenarios
- Content designed to frighten, distress, or psychologically harm a child
- Hate speech or slurs

### Post-Generation Check

After the story text is generated, a second safety check reviews the full story before it is assembled into the HTML. If the story contains any of the prohibited content types despite input screening, the story is discarded, and the user is shown a message saying: **"Something went wrong generating your story — please try again or change your inputs."** The failed generation is not saved to the library.

### What Happens When Content Is Blocked

If any input field triggers a safety block during the screening phase, generation does not start. The user sees a clear, non-judgmental error message on the form. The message identifies which field is problematic and offers it for re-entry.

**Example:** "The story topic you entered isn't quite right for a children's story — could you describe it differently?"

**Tone requirement:** The error message must never be condescending or accusatory — many blocks will be accidental (a user typing "dragon kills the villain" not realizing violence is excluded).

## Story Library / History

### Every Story Is Saved Automatically

When a story is successfully generated and passes safety checks, it is saved to the local library **before** the user is shown the storybook. There is no "save" button. The user should never lose a story because they forgot to save it.

### What Is Stored Per Story

For each story, the following is stored in the local library:

- The complete assembled HTML file for the storybook
- All generated illustration images (either embedded as base64 in the HTML or stored in a sibling `images/` directory)
- **Metadata:** title, child's name (if provided), age tier, date generated, story topic (first 100 chars), cover thumbnail image

The metadata is stored in a simple local JSON index file that the application reads on startup. The HTML and images are stored in a folder named after the story's unique ID (e.g., `story_<uuid>/`).

### The Library View

The home page displays the library as a **visual shelf**. Each story is represented by its cover thumbnail, title, and the child's name (if stored). Stories are ordered by most recently generated, newest first. The shelf displays a maximum of 20 stories before paginating (simple prev/next buttons). Clicking a story opens it directly to the cover page, exactly as it looked when first generated.

### Deleting Stories

A story can be deleted from the library. The user must confirm deletion with a simple "Are you sure?" modal. Deleting a story removes:

- Its HTML file
- Its images directory
- Its metadata entry from the JSON index

There is no recycle bin or soft delete.

### No Editing After Generation

There is no story editing feature within the application. If the user wants a different version, they generate a new story with revised inputs.

## Local-First Principle

### Why Local-Only

This application runs entirely on the user's machine. No story content, no child's name, no illustrations, and no usage data ever leave the device. This is not a privacy policy — it is a technical architecture constraint. There is no backend server, no cloud inference endpoint, no analytics SDK, no CDN.

For a product used with children's personal information and a child's likeness preferences, local-only is the only ethical architecture for a personal tool.

### What Local-Only Implies

- The application is a Python process that serves a local web server (accessible at `localhost`). The user opens a browser and navigates to that address.
- All model inference runs on the same machine.
- Generated stories are stored on the local filesystem.
- The application works with no internet connection once the models are downloaded.
- The models must be downloaded once during a setup step. After setup, the application must be entirely offline-capable.
- The application should fail gracefully and clearly if a required model is not found locally. It must never silently attempt a cloud fallback.

### Model Selection Philosophy

The local LLM for text generation must be capable of following detailed multi-constraint prompts reliably. The local image generation model must produce cartoon-style, child-appropriate imagery consistently. Specific model versions are implementation decisions, but the PRD mandates: **the user must never have to choose, configure, or interact with the models directly.** Model selection and parameter tuning is the developer's responsibility, not the user's.

## Output Format

### The Storybook Is a Self-Contained HTML File

The final storybook for each story is a **single self-contained HTML file**. All CSS is inline or in a `<style>` block within the file. All JavaScript for the page-turn animation is inline in a `<script>` block. All illustrations are either embedded as base64-encoded data URIs or referenced from a sibling `images/` directory. The HTML file must render correctly by opening it directly in any modern browser — it must not require the application server to be running in order to read.

### HTML Structure Is Fixed and Versioned

The HTML template used to assemble the storybook must not change from story to story or from generation to generation. It is a **fixed template into which content is injected.** This ensures:

- Visual consistency across all stories in the library
- The ability to regenerate or re-render a story without changing its structure
- Predictability for the developer when making UI changes (a change to the template affects all stories uniformly)

The template version is embedded as a data attribute on the root element: `data-template-version="1.0"`. If the template ever changes in a breaking way, the version increments and the library handles old and new versions gracefully.

### Page Structure Within the HTML

Each page in the storybook is a `<div>` with a consistent class structure:

- A wrapper div representing the page with its background and theme classes
- An illustration container (empty for pages without illustrations)
- A text container with the page's story text
- A page number element
- A `data-page-type` attribute (cover, dedication, body, final) that drives CSS targeting

The page-turn animation operates on these divs using CSS 3D transforms. The JavaScript layer is minimal — it tracks current page index and applies/removes animation classes. **No JavaScript framework is used in the storybook HTML.** It must work in plain HTML/CSS/JS.

## Non-Goals

The following are explicitly outside the scope of this product. Any feature request that falls into this list should be declined without guilt.

**Not a cloud application.** There is no hosted version, no user accounts, no login, no subscription. This will never be monetized or served to users other than the person running it locally.

**Not a story editor.** Once a story is generated, the text cannot be edited within the application. There is no rich text editor, no page reordering, no character name find-and-replace. Regeneration is the editing workflow.

**Not a multi-language tool.** All stories are generated in English. Internationalization and locale support are out of scope for v1.

**Not a collaborative tool.** There is one user — the person running the local server. There are no sharing features, no collaborative editing, no "send to grandparent" feature.

**Not a voice narration tool.** Text-to-speech narration is not part of this product. There is no audio playback, no read-aloud mode.

**Not a printing tool.** The storybook is designed for screen reading. Print stylesheet support and physical printing are out of scope.

**Not customizable by end users.** Users cannot change fonts, color themes, illustration style, page count, or template layout. All of those are locked design decisions made by the developer.

**Not a general-purpose story generator.** This tool generates children's stories only, for ages 2–10 only, in the storybook format only. Adult stories, educational content, interactive fiction, and other narrative formats are out of scope.

## Quality Bar

### Quality Bar for a Story

A story meets the quality bar when all of the following are true:

- A first-time reader of the story cannot tell it was machine-generated — it reads as if a thoughtful human wrote it for this specific child.
- The moral or lesson is clearly present and feels organic to the story, not tacked on.
- The child's name (if provided) appears naturally in the narrative, not awkwardly inserted or in parentheses.
- Every page's text is self-contained enough that a reader can understand it in context without seeing previous pages, but the narrative flows clearly from page to page.
- The vocabulary and sentence complexity are appropriate for the specified age — the developer should test by having a target-age child (or an adult simulating one) attempt to follow the story.
- No grammatical errors, no broken sentences, no unintended repeated phrases.
- The page count matches the tier specification (Tier 1: 6–8 body pages; Tier 2: 10–12; Tier 3: 12–16).

### Quality Bar for an Illustration

An illustration meets the quality bar when all of the following are true:

- The main character depicted on the page is visually consistent with their depiction on all other pages — same color scheme, same style, same proportions.
- The illustration's content matches the text on the same page — it illustrates the specific scene, not a generic scene.
- The style reads as cartoon/illustrated, not photorealistic.
- The illustration contains no visible text, watermarks, artifacts, or distortions.
- The illustration is appropriate for children — no frightening imagery, no graphic content, no adult themes visible.
- The illustration is aesthetically pleasing — a parent looking at it thinks "this looks like a real picture book," not "this looks like AI made it."

### Quality Bar for the Overall Experience

The overall experience meets the quality bar when:

- A non-technical adult can complete the entire flow — from opening the application to reading a finished story — without asking for help or reading documentation.
- The generation process never leaves the user wondering if the application has crashed — progress is always communicated via status messages.
- The page-turn animation feels smooth and physical on a standard laptop from the last 5 years.
- Opening the saved HTML file directly in a browser (without the server running) produces a correctly rendered, fully readable storybook.
- Two different stories generated with the same template feel visually familiar (same structure, fonts, animations) while looking thematically distinct (different palettes, different illustration content).
- After generating 10 stories, the library is navigable and pleasant to browse — it feels like a shelf of books, not a list of files.

## Guiding Questions for Design Decisions

When the developer faces a design decision not explicitly covered by this PRD, the following questions should resolve it:

1. **Would a parent reading this story to a 5-year-old at bedtime feel comfortable with it?** If uncertain, the answer is no.
2. **Does this feature require the user to understand anything about AI, models, or technical concepts?** If yes, it violates the local-first, user-simplicity principle.
3. **Does this add a new input, setting, or choice for the user?** Every addition must be justified against the principle that simpler inputs produce a better, more focused experience.
4. **Does this change the HTML template in a way that breaks existing stories?** If yes, the template version must be incremented and backward compatibility must be handled.
5. **Does this feature require internet access or data leaving the device?** If yes, it is out of scope.
